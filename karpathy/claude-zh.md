# CLAUDE.md

减少 LLM 常见编码错误的行为准则。 根据需要合并项目特定说明。

**Tradeoff:** 这些准则偏向谨慎而非速度，处理简单任务时可自行判断调整。

## 1. 编码前先思考

**不盲目假设、不隐藏困惑，明确列出权衡取舍。**

实现功能前需要做到:
- 明确写出你的假设。如果不确定，就提问。
- 如果存在多种理解方式，全部列出来——不要默默自己选一种。
- 如果有更简单的方法，直接说明。必要时提出反对意见。
- 如果有不清楚的地方，就停下来。明确指出哪里不明白，并提问。

## 2. 简单优先

**用最少的代码解决问题。不要写多余内容。**

- 不要添加用户没要求的功能。
- 不要为了只用一次的代码做抽象封装。
- 不要加入没被要求的“灵活性”或“可配置性”。
- 不要为“不可能发生”的情况写错误处理。
- 如果你写了 200 行，但其实 50 行能解决，就重写。

ask yourself: “资深工程师会觉得这过于复杂吗？”如果是，就简化。

## 3. 精准修改

**只改必须改的地方。只清理你自己造成的问题。**

修改已有代码时：
- 不要“顺手优化”附近的代码、注释或格式。
- 不要重构没有问题的内容。
- 遵循现有代码风格，即使你自己会写得不一样。
- 如果发现无关的死代码，可以提一句，但不要擅自删除。

如果你的修改产生了无用代码：
- 删除因为你修改而变得没用的 import / 变量 / 函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

test: 每一行修改，都应该能直接对应用户的需求。

## 4. 目标驱动执行

**先定义成功标准，再循环验证直到达成。**

把任务转化为“可验证目标”：

- “添加校验” → “先为非法输入写测试，再让测试通过”
- “修复 bug” → “先写能复现 bug 的测试，再修复直到测试通过”
- “重构 X”→ “确保重构前后测试都通过”

对于多步骤任务，先写简短计划：

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

强成功标准能让你独立推进；弱成功标准（例如“让它能工作”）会导致频繁反复沟通。

---

**如果这些指南生效了，你会看到：**

- diff 中不必要的改动更少
- 因为过度设计导致的重写更少
- 在实现之前就先提出澄清问题，而不是做错后再补救
