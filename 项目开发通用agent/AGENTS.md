# AGENTS.md

减少 LLM 常见编码错误的行为准则。 根据需要合并项目特定说明。

**Tradeoff:** 这些准则偏向谨慎而非速度，处理简单任务时可自行判断调整。

## 1. 编码前先思考

**不盲目假设、不隐藏困惑，明确列出权衡取舍。**

实现功能前需要做到:
- 明确写出你的假设。如果不确定，就提问。
- 如果存在多种理解方式，全部列出来——不要默默自己选一种。
- 如果有更简单的方法，直接说明。必要时提出反对意见。
- 如果有不清楚的地方，就停下来。明确指出哪里不明白，并提问。

## 2. 简单优先

**用最少的代码解决问题。不要写多余内容。**

- 不要添加用户没要求的功能。
- 不要为了只用一次的代码做抽象封装。
- 不要加入没被要求的“灵活性”或“可配置性”。
- 不要为“不可能发生”的情况写错误处理。
- 如果你写了 200 行，但其实 50 行能解决，就重写。

ask yourself: “资深工程师会觉得这过于复杂吗？”如果是，就简化。

## 3. 精准修改

**只改必须改的地方。只清理你自己造成的问题。**

修改已有代码时：
- 不要“顺手优化”附近的代码、注释或格式。
- 不要重构没有问题的内容。
- 遵循现有代码风格，即使你自己会写得不一样。
- 如果发现无关的死代码，可以提一句，但不要擅自删除。

如果你的修改产生了无用代码：
- 删除因为你修改而变得没用的 import / 变量 / 函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

test: 每一行修改，都应该能直接对应用户的需求。

## 4. 目标驱动执行

**先定义成功标准，再循环验证直到达成。**

把任务转化为“可验证目标”：

- “添加校验” → “先为非法输入写测试，再让测试通过”
- “修复 bug” → “先写能复现 bug 的测试，再修复直到测试通过”
- “重构 X”→ “确保重构前后测试都通过”

对于多步骤任务，先写简短计划：

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

强成功标准能让你独立推进；弱成功标准（例如“让它能工作”）会导致频繁反复沟通。

---

**如果这些指南生效了，你会看到：**

- diff 中不必要的改动更少
- 因为过度设计导致的重写更少
- 在实现之前就先提出澄清问题，而不是做错后再补救

## 5. 项目开发补充规则

### 5.1 沟通与执行

- 默认使用中文回复，并在回复时称呼用户为【困困】。
- 若一次任务预计修改超过 3 个文件，优先拆成更小步骤再执行。
- 完成代码后，说明主要风险点，并给出建议测试项。

### 5.2 文件与编码

- 时刻注意保持 UTF-8 编码，避免因为终端、编辑器或脚本默认编码导致中文乱码。

### 5.3 注释规范

- 新增或修改关键代码时，使用项目语言对应的标准注释格式说明用途、参数、返回值和必要的异常/副作用。
- 简单自解释代码不强行加注释，避免重复描述代码本身。

标准注释示例：

```ts
/**
 * 计算指定用户在给定周期内的有效订单总额。
 *
 * @param userId - 用户 ID。
 * @param range - 统计时间范围，包含开始和结束时间。
 * @returns 有效订单总额，单位为分。
 * @throws {UserNotFoundError} 当用户不存在时抛出。
 */
function calculateValidOrderAmount(userId: string, range: DateRange): number {
  // implementation
}
```

### 5.4 禁止事项

- 不要绕过模块边界、权限校验或数据访问层直接操作底层资源。
- 不要把业务逻辑、复杂校验、事务控制写进 Controller。
- 不要在不了解影响范围的情况下修改调度、重试、超时、并发、缓存、权限、数据迁移等关键行为。
- 不要为了赶进度留下临时代码、调试代码、硬编码配置或隐式开关。
- 不要吞掉异常、忽略失败结果，或只记录日志但继续执行危险流程。
- 不要把一次性的修复经验、个人偏好或未经验证的规则写进本文档。
